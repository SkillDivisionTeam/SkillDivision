## 2. Рамки проекта

**Временные рамки:**

- Дата начала проекта: 17 октября 2025 года.
- Планируемая дата выпуска MVP: 10 декабря 2025 года
- Окончание первой фазы развития: 31 мая 2026 года

**Функциональные рамки:**

- Telegram-бот для регистрации участников, проведения квизов и отображения базовой статистики.
- Веб-интерфейс для организаторов (создание мероприятий, генерация вопросов с ИИ) и HR (анализ данных, экспорт профилей в CSV/PDF).
- Базовая аналитика (Skill Division, реал-тайм мониторинг).

**Ресурсные рамки:**

- Команда из 5 человек
- Хостинг: Heroku/Vercel или Yandex Cloud с лимитом до $100/месяц.
- Инструменты: Telegram API, веб-фреймворки (Next.js, FastAPI), базы данных (PostgreSQL).
- ИИ: Использование бесплатных/пробных API (Yandex GPT, GigaChat) с переходом на платные при масштабировании.

### 2.1. Матрица компромиссов проекта

В рамках нашего проекта приоритеты распределяются следующим образом:

| **Параметр**    | **Фиксируется** | **Согласовывается** | **Принимается** |
| --------------- | --------------- | ------------------- | --------------- |
| **Ресурсы**     | ✅               |                     |                 |
| **Время**       |                 | ✅                   |                 |
| **Возможности** |                 |                     | ✅               |

**Основания выбора:**

- **Ресурсы фиксируются,** так как команда проекта состоит из пяти участников с четко распределенными ролями и ограниченным бюджетом. Увеличение количества членов команды или бюджета невозможно без утверждения заказчика.
- **Время согласовывается,** поскольку проект имеет заранее установленные ключевые вехи (выпуск MVP и первая демонстрация), но допустимо изменение сроков отдельных этапов в пределах разумного.
- **Возможности принимаются,** то есть функциональность решения может быть адаптирована под рамки доступных ресурсов и сроков. В случае нехватки времени или бюджета часть функций может быть реализована в следующей фазе.

Таким образом, приоритеты устанавливаются в пользу стабильности команды и бюджета, с возможностью корректировки объема функционала и частичных сдвигов сроков.

## 4. Протоколы проекта

### 4.1. Управление конфигурацией

**1\. Методы и средства управления конфигурацией:**

- **Система контроля версий:** Git с хостингом на GitHub
  - **Модель ветвления:** Используется GitFlow или упрощенная модель с главными ветками:
  - **main** — стабильная версия, соответствующая последнему развернутому релизу.
  - **develop** — основная ветка для интеграции новых функций.
  - **feature/\*** — ветки для разработки новых функций, ответвляются от develop и вливаются в нее после код-ревью.
  - **hotfix/\*** — ветки для срочных исправлений в production, ответвляются от main и вливаются обратно в main и develop.
- **Среды развертывания:**
  - **Development (Разработка):** Локальные машины разработчиков.
  - **Staging (Тестирование):** Средой, максимально приближенная к продакшену, используется для приемочного тестирования и проверки сборки.
  - **Production (Продакшен):** Финальная среда для конечных пользователей.
  - Для унификации окружений и устранения различий между машинами разработчиков, все основные сервисы проекта (backend, frontend, Telegram-бот и база данных PostgreSQL) разворачиваются в контейнерах **Docker**.
  - Сборка и запуск локальных и тестовых сред осуществляется через **Docker Compose**, что обеспечивает идентичность конфигурации между этапами разработки, тестирования и продакшена.
- **Документация:**
  - Используется **Google Docs** для хранения требований, диаграмм и технической документации.
  - Все документы хранятся в общей папке на **Google Drive**, обеспечивающей централизованный доступ для команды.
  - Доступ к документам предоставляется участникам проекта в соответствии с их ролями (чтение/редактирование).

**2\. Процесс запроса и принятия изменений в конфигурации:**

- **Запрос:** Любой участник команды создает issue в GitHub с описанием предлагаемого изменения (новый функционал, баг, улучшение).
- **Обсуждение:** Предложение обсуждается командой.
- **Реализация:** После одобрения, создается feature-ветка, в которой ведется разработка.
- **Код-ревью:** После завершения разработки создается Pull Request (PR) из feature/\* в develop. Обязательное прохождение код-ревью как минимум одним другим разработчиком.
- **Тестирование:** После мержа PR в develop, код автоматически развертывается на staging-среду для тестирования.
- **Релиз:** После успешного тестирования, создается PR из develop в main для формирования нового релиза.

**3\. Роли и ответственности:**

- Разработчики: Создание feature-веток, ведение разработки, создание PR.
- Техлид: Проведение код-ревью, утверждение мержа PR в develop.
- Менеджер проекта: Приоритизация проблем, утверждение крупных функциональных изменений.
- Тестировщик: Проведение тестирования на staging-среде и подтверждение готовности к релизу.

**4\. Требования к системе контроля версий:**

- Все коммиты должны содержать понятные сообщения.
- Запрещен прямой push в ветки main и develop.
- Мерж в develop и main возможен только через Pull Request с прохождением код-ревью.
- Код в ветке main должен всегда быть стабильным и готовым к развертыванию.

### 4.2. Управление изменениями

**1\. Процесс управления изменениями:**

Процесс управления изменениями формализует внесение правок в утвержденные ранее артефакты: ТЗ, дизайн, рамки проекта.

1. Инициация: Заинтересованная сторона (заказчик, член команды) заполняет Форму запроса на изменение (см. ниже).
2. Анализ: Менеджер проекта вместе с техническим лидом анализирует запрос. Оценивается влияние на треугольник компромиссов: сроки, ресурсы, рамки. Результатом анализа является рекомендация (принять/отклонить/принять с изменениями).
3. Решение: Руководство в составе Менеджера проекта, Технического лида и Представителя заказчика рассматривает запрос и анализ, принимает финальное решение.
4. Реализация: Если изменение утверждено, Менеджер проекта обновляет бэклог продукта, план проекта и инициирует процесс реализации через систему управления конфигурацией.

**2\. Форма запроса на изменение:**

- Идентификатор запроса: [CR-001]
- Дата подачи: [Дата]
- Инициатор: [Имя, Роль]
- Краткое описание изменения: [1-2 предложения]
- Обоснование изменения: [Почему это изменение необходимо? Какая проблема решается?]
- Приоритет: [Высокий/Средний/Низкий]
- Анализ воздействия (заполняется МП/Техлидом):
- Влияние на сроки: [+0 дней / +X недель]
- Влияние на ресурсы/бюджет: [+0 у.е. / +X у.]
- Влияние на рамки/функциональность: [Какие функции добавляются/изменяются/удаляются?]
- Влияние на риски: [Какие новые риски возникают?]
- Рекомендация: [Принять / Отклонить / Принять с условиями]
- Решение Control Board: [Утверждено / Отклонено] [Дата] [Подпись]

**3\. Роли и ответственности:**

- Инициатор: Заполняет форму запроса.
- Менеджер проекта: Координирует процесс анализа, ведет протокол заседаний Control Board.
- Технический лид: Проводит технический анализ воздействия.
- Control Board: Принимает финальное решение по запросу.

**4\. Действия при влиянии на контракт:**

Любое изменение, существенно влияющее на сроки, бюджет или объем работ (выходящее за рамки, утвержденные в "Критериях одобрения решения"), считается изменением контракта. В этом случае утвержденное руководством изменение формализуется в виде Дополнительного соглашения к контракту, которое подписывается уполномоченными представителями заказчика и исполнителя перед началом реализации изменения.

### 4.3. Управление внедрениями

**Стратегия внедрения:**

Внедрение будет проводиться итеративно, начиная с пилотного запуска, как указано в "Задачах проекта".

**Мероприятия по внедрению:**

**Подготовка релиза:**

1. Создание ветки релиза от develop.
2. Обновление версии, документации, релизных заметок (changelog).
3. Финальное тестирование на staging-среде, включая проверку критериев приемки.
4. Создание и тестирование скриптов для миграции базы данных (если необходимо).

**Пилотное внедрение (Первый релиз):**

1. Цель: Проверить работу системы в реальных условиях, собрать обратную связь.
2. Масштаб: 1-2 митапа, ~50 участников (как указано в "Предположениях").
3. Действия: Развертывание на production-среде, мониторинг работы, сбор метрик и отзывов от организаторов и участников.

**Финальное внедрение (После успеха пилота):**

1. Обработка обратной связи, устранение критических багов.
2. Плановый релиз с доработанным функционалом.
3. Широкое представление решения IT-сообществу.

**Откат (Rollback Plan):**

1. Определены ключевые точки отката (например, при падении сервера более 10 минут).
2. Подготовлена резервная копия базы данных перед каждым релизом.
3. Возможность быстро развернуть предыдущую стабильную версию из ветки main.

  - Принятие решений о готовности к релизу

#### 4.4.5. Методологии и подходы к обеспечению качества

Проект реализуется с использованием **инкрементальной модели разработки** с короткими итерациями (спринтами). Каждый спринт завершается демонстрацией функционала и сбором обратной связи, что позволяет раннее выявление и устранение дефектов.

Для контроля архитектурного качества и чистоты кода применяется набор методологий и паттернов:

- **Архитектурные паттерны:**
  - **MVC (Model–View–Controller)** — реализован в серверной части (Django).
  - **Client–Server** — Telegram-бот и веб-интерфейс работают как клиенты REST API.
  - **Thin Client** — бизнес-логика сосредоточена на сервере, что облегчает тестирование и контроль версий.
  - **Repository / ORM** — отделение логики данных от бизнес-логики через Django ORM.
- **Методологии обеспечения качества кода:**
  - **Code Review** — обязательная проверка всех pull request перед мержем.
  - **Continuous Integration (CI)** — автоматическое тестирование при каждом изменении в develop-ветке.
  - **Static Code Analysis** — использование линтеров (flake8, pylint) для предотвращения технического долга.

### 4.5. Рабочая среда проекта

Все члены команды работают удаленно из личных рабочих мест (дома или коворкингов). Гибкий график с пересечением рабочего времени минимум 4 часа в день. Основной канал связи — discord или Telegram для оперативного общения. Команда самостоятельно обеспечивает эргономичные рабочие места (стол, стул), соответствующие стандартам длительной работы

Каждый член команды оснащен персональным компьютером или ноутбуком с процессором не ниже Intel i5/Ryzen 5, 16/32 ГБ ОЗУ и SSD на 512/1024 ГБ для обеспечения производительности при разработке, тестировании и анализе данных. Минимальная скорость интернета — 50 Мбит/с для загрузки и 20 Мбит/с для выгрузки.
